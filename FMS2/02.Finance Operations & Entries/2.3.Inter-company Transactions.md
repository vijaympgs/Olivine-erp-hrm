# 2. PURPOSE

## 2.1 Business Purpose
Inter-company Transactions module enables organizations to manage financial transactions between different legal entities within the same corporate group. This module facilitates proper accounting treatment, elimination of inter-company profits/losses, and ensures compliance with consolidation requirements for multi-entity organizations.

## 2.2 Business Scope
- Inter-company invoice and credit note processing
- Inter-company loan and interest management
- Expense allocation and cost sharing between entities
- Inter-company service charges and fee processing
- Currency conversion for cross-border inter-company transactions
- Automated inter-company reconciliation and matching
- Elimination entries for consolidation purposes
- Regulatory compliance for transfer pricing documentation

## 2.3 Key Stakeholders
- **Group Finance Teams:** Manage inter-company policies and consolidation
- **Entity Finance Managers:** Process inter-company transactions and reconciliations
- **Tax Teams:** Handle transfer pricing documentation and compliance
- **Auditors:** Review inter-company transactions and elimination entries
- **Management:** Monitor inter-company balances and cash flows

class InterCompanyTransaction(models.Model):
    """
    Main model for managing inter-company financial transactions

    id = models.UUIDField(primary_key=True, default=uuid.uuid4, editable=False)

    company_id = models.UUIDField()

    currency_id = models.UUIDField()

    # Core Fields
    transaction_number = models.CharField(max_length=30, unique=True)
    transaction_date = models.DateField()
    description = models.CharField(max_length=200)
    amount = models.DecimalField(max_digits=20, decimal_places=2)
    exchange_rate = models.DecimalField(max_digits=20, decimal_places=10, default=1.0)
    base_currency_amount = models.DecimalField(max_digits=20, decimal_places=2)
    due_date = models.DateField(null=True, blank=True)
    reference_number = models.CharField(max_length=50, blank=True)
    status = models.CharField(max_length=20, choices=[
        ('DRAFT', 'Draft'),
        ('PENDING', 'Pending Approval'),
        ('APPROVED', 'Approved'),
        ('POSTED', 'Posted'),
        ('REVERSED', 'Reversed'),
        ('CANCELLED', 'Cancelled'),
    ], default='DRAFT')
    transfer_pricing_method = models.CharField(max_length=50, blank=True)
    arm_length_indicator = models.BooleanField(default=True)
    
    # Audit Fields
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)
    
    class Meta:
        db_table = 'fms_inter_company_transactions'
        verbose_name = 'Inter-company Transaction'
        verbose_name_plural = 'Inter-company Transactions'
        indexes = [
            models.Index(fields=['company_id', 'from_entity_id', 'to_entity_id'], name='idx_intercompany_entities'),
            models.Index(fields=['company_id', 'transaction_date'], name='idx_intercompany_date'),
            models.Index(fields=['status'], name='idx_intercompany_status'),
            models.Index(fields=['transaction_number'], name='idx_intercompany_number'),
        ]

            models.UniqueConstraint(fields=['company_id', 'transaction_number'], name='uk_intercompany_transaction_number'),

# 6. BUSINESS RULES & VALIDATIONS

## 6.1 Data Integrity Rules
- From and to entities cannot be the same in a transaction
- Transaction amounts must be positive
- Due date must be after transaction date
- ARM length indicator must be documented for transfer pricing compliance
- Inter-company transactions must have proper approval workflow

## 6.2 Validation Logic
```python
def clean_entities(self):
    """
    Validate that from and to entities are different
    """
    if self.from_entity_id == self.to_entity_id:
        raise ValidationError("From and to entities cannot be the same")

def clean_amount(self):
    """
    Validate transaction amount
    """
    if self.amount <= 0:
        raise ValidationError("Transaction amount must be greater than zero")

def clean_due_date(self):
    """
    Validate due date against transaction date
    """
    if self.due_date and self.due_date < self.transaction_date:
        raise ValidationError("Due date cannot be before transaction date")
```

## 6.3 Business Constraints
- Each entity pair can have only one pending transaction for the same period
- Reconciliation variance amounts must be explained and approved
- Transfer pricing documentation must be maintained for tax compliance
- Inter-company eliminations must be generated during consolidation

# 9. API SPECIFICATIONS

## 9.1 REST Endpoints

#### **GET /api/inter-company-transactions/**
- **Purpose:** Retrieve all inter-company transactions for the company
- **Authentication:** JWT token required
- **Parameters:** company_id, from_entity, to_entity, status, date_range
- **Response:** List of inter-company transaction objects

#### **POST /api/inter-company-transactions/**
- **Purpose:** Create new inter-company transaction
- **Authentication:** JWT token required
- **Request Body:** Transaction details with line items
- **Response:** Created transaction object with ID

#### **POST /api/inter-company-transactions/{id}/approve/**
- **Purpose:** Approve inter-company transaction
- **Authentication:** JWT token required
- **Request Body:** Approval details and comments
- **Response:** Updated transaction status

#### **POST /api/inter-company-reconciliation/**
- **Purpose:** Trigger automatic reconciliation process
- **Authentication:** JWT token required
- **Request Body:** Reconciliation parameters and entity pairs
- **Response:** Reconciliation job ID with status tracking

#### **GET /api/inter-company-entities/**
- **Purpose:** Retrieve all legal entities for inter-company transactions
- **Authentication:** JWT token required
- **Parameters:** company_id, is_active
- **Response:** List of entity objects with relationships

## 9.2 Data Formats
- **Request Format:** JSON with proper decimal precision for financial amounts
- **Response Format:** JSON with standardized transaction and entity data
- **Error Format:** JSON with error code, message, and validation details
