# 5.7.6 REPLENISHMENT SUGGESTIONS

**Type**: OPERATIONAL MODULE  
**Status**: DRAFTING  
**Owner**: Astra  

---

## 1. üéØ OBJECTIVE
Generate actionable purchasing advice based on current stock and Reorder Policies. This is the "ToDo List" for buyers.

## 2. üë• USERS
*   **Purchasing Agent**: Review and convert suggestions to Purchase Requisitions.

## 3. üõ†Ô∏è REQUIREMENTS

### 3.1 Functional
*   **Calculation Engine**: Run periodically or on-demand.
    *   Logic: `IF (Available_Stock + On_Order) < Reorder_Point THEN Suggest_Order`.
    *   Qty Calculation: 
        *   Fixed: `Reorder_Qty`
        *   Min-Max: `Max_Qty - (Available + On_Order)`.
*   **Approval**: Buyer reviews suggestion -> Edits Qty -> Creates PR.

### 3.2 Data Fields
*   **Source**: Item, Location.
*   **Trigger**: Current Stock vs Reorder Point.
*   **Suggestion**: Suggested Qty, Vendor.
*   **Status**: Suggested, Ignored, Ordered.

### 3.3 Actions
*   **Create PR**: Bulk action to generate Purchase Requisitions (Integration with Procurement).
*   **Ignore**: Dismiss suggestion for now.

## 4. üìä DATA MODEL
*   **Input**: `Inventory`, `StockMovement` (On Order), `ReorderPolicy`.
*   **Output**: Transient view or temporary `ReplenishmentSuggestion` objects (in memory or simple table).
*   **Target**: `PurchaseRequisition` (Procurement Module).

## 5. üñ•Ô∏è UI MOCKUP
*   **Screen**: `/inventory/replenishment/worksheet`
*   **Layout**:
    *   List of suggestions grouped by Vendor.
    *   Columns: Item, Current Stock, Suggestion, Vendor.
    *   Action: "Generate Orders".

## 6. ‚úÖ ACCEPTANCE CRITERIA
1.  Correctly identifies items needing reorder.
2.  Considers "On Order" (Purchase Orders not yet received) to avoid double ordering.
3.  Generates valid Purchase Requisitions.
