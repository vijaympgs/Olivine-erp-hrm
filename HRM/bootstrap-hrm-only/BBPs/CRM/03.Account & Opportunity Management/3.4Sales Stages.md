## **2. PURPOSE**

### **2.1 Business Purpose**
The Sales Stages module serves as the master configuration system for defining and managing sales process stages within the CRM. It provides comprehensive tools for creating, configuring, and maintaining sales stage definitions that govern opportunity progression, probability calculations, and workflow automation. The module enables sales operations teams to design customized sales processes, implement stage-specific business rules, and ensure consistent sales methodology execution across the organization.

### **2.2 Business Scope**

**In Scope:**
- Complete sales stage lifecycle management with customizable stage definitions and configurations
- Stage transition rules and validation logic with automated workflow enforcement
- Probability management with default probabilities and override capabilities
- Stage duration tracking with average and maximum duration monitoring
- Required field definitions and validation rules for each sales stage
- Stage-specific approval workflows and exception handling processes
- Multi-stage process design with branching and conditional logic support
- Integration with opportunity pipeline for automated stage progression
- Stage performance analytics and conversion rate tracking
- Mobile-responsive interface for stage management and configuration

**Out of Scope:**
- Opportunity creation and management (handled by Opportunity Pipeline module)
- Account and contact data management (handled by Account Management and Contact Directory modules)
- Quote generation and pricing management (handled by Sales & Quote Management modules)
- Advanced financial calculations and billing (handled by external ERP systems)
- Customer service and support case management (handled by Customer Service & Support modules)

### **2.3 Key Stakeholders**
- **Sales Operations Teams:** Primary users for designing, configuring, and maintaining sales stage definitions
- **Sales Managers:** Monitor stage performance, conversion rates, and provide feedback on process optimization
- **Sales Directors:** Oversee sales process design, methodology implementation, and strategic stage planning
- **System Administrators:** Manage technical configurations, integrations, and system maintenance
- **Sales Representatives:** Follow defined sales stages and provide feedback on process usability
- **Executives:** Review sales process effectiveness and strategic alignment with business goals

- **StageTransition** → **SalesStage (to_stage_id):** Many-to-One relationship defining the destination stage in transitions

### **5.2 Many-to-Many Relationships**

- **SalesStage** ↔ **StageCategory:** Many-to-Many relationship through stage_category associations for categorization
- **SalesStage** ↔ **StageAction:** Many-to-Many relationship through stage_action associations for automated actions
- **StageTransition** ↔ **StageAction:** Many-to-Many relationship through transition_action associations for transition-specific actions

### **5.3 Relationship Integrity Rules**

- **Cascade Delete:** When a stage is deleted, all associated transitions are soft-deleted but history is preserved
- **Protect Delete:** Company and user records referenced in stages are protected from deletion to maintain data integrity
- **Restrict Delete:** Stages referenced in active opportunities are restricted from deletion to maintain pipeline consistency
- **Self-Referential Integrity:** Stage transitions cannot create circular references or invalid stage sequences

### **5.4 Stage Configuration Constraints**

- **Stage Order Uniqueness:** Stage order must be unique within each company to prevent conflicts
- **Transition Validation:** Transitions can only be created between valid stage combinations without circular references
- **Probability Consistency:** Default probabilities must follow logical progression patterns across stages
- **Closure Stage Rules:** Only one stage can be marked as closed-won and one as closed-lost per company

## **6. BUSINESS RULES & VALIDATIONS**

### **6.1 Data Integrity Rules**

- **Unique Stage Code:** Each stage must have a unique stage code within the company to prevent duplicates
- **Required Fields:** Stage name, stage code, and stage order are mandatory fields for all sales stages
- **Stage Order Validation:** Stage order must be sequential and unique within each company
- **Probability Range:** Default probability must be between 0 and 100 and follow logical progression
- **Closure Stage Rules:** Only one stage can be marked as closed-won and one as closed-lost per company

### **6.2 Validation Logic**

```python
def clean_stage_order(self):
    """
    Validate stage order uniqueness and sequential progression
    """
    if self.stage_order is not None:
        # Check for duplicate stage order within company
        if SalesStage.objects.filter(
            company_id=self.company_id,
            stage_order=self.stage_order,
            is_deleted=False
        ).exclude(id=self.id).exists():
            raise ValidationError("Stage order must be unique within the company")

def clean_default_probability(self):
    """
    Validate probability range and logical progression
    """
    if self.default_probability is not None:
        if not (0 <= self.default_probability <= 100):
            raise ValidationError("Default probability must be between 0 and 100")
        
        # Check logical progression based on stage order
        if self.stage_order > 1:
            previous_stage = SalesStage.objects.filter(
                company_id=self.company_id,
                stage_order=self.stage_order - 1,
                is_deleted=False
            ).first()
            if previous_stage and self.default_probability < previous_stage.default_probability:
                raise ValidationError("Default probability should not decrease with stage progression")

def clean_stage_code(self):
    """
    Validate stage code format and uniqueness
    """
    if not self.stage_code:
        raise ValidationError("Stage code is required")
    
    # Check for valid format (alphanumeric with underscores)
    if not re.match(r'^[A-Za-z0-9_]+$', self.stage_code):
        raise ValidationError("Stage code can only contain letters, numbers, and underscores")
    
    # Check uniqueness within company
    if SalesStage.objects.filter(
        company_id=self.company_id,
        stage_code=self.stage_code,
        is_deleted=False
    ).exclude(id=self.id).exists():
        raise ValidationError("Stage code must be unique within the company")
```

### **6.3 Business Constraints**

- **Stage Deletion:** Stages cannot be deleted if they are referenced in active opportunities or transitions
- **Transition Validation:** Transitions cannot create circular references or invalid stage sequences
- **Closure Stage Modification:** Closed stages cannot be modified to non-closed status if referenced in opportunities
- **Probability Overrides:** Manual probability overrides require appropriate authorization and justification

- **Purpose:** Create new sales stage with validation and automatic ordering
- **Authentication:** JWT token with create_stage permission
- **Request Body:** Sales stage data with required fields validation
- **Response:** Created sales stage object with validation warnings and calculated order

#### **GET /api/sales-stages/{id}/**
- **Purpose:** Retrieve detailed sales stage information including transitions and actions

- **Authentication:** JWT token with update_stage permission

- **Response:** Updated sales stage object with change summary
